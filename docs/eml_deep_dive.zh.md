# EML 深潜：一个算子统治一切

> TL;DR: 有个数学家证明，`exp(x) - ln(y)` 加上常量 `1`，就足以造出你 TI-84 计算器上的每一个按钮。这篇文档告诉你为什么这事重要、这个发现到底是怎么做出来的（剧透： brute force），以及把它写成能跑的 TypeScript 长什么样。

---

## 1. 一句话说完

你知道 NAND 吧？一个门。把自己喂给自己足够多次，你就能得到 AND、OR、NOT、XOR、加法器、CPU、整棵技术树。

Andrzej Odrzywolek 问了一个问题：_连续数学里有没有 NAND？_

答案是有。就是这个看起来有点怪的二元算子：

```
eml(x, y) = exp(x) - ln(y)
```

只需要 `1`，然后反复用 `eml`，你就能还原出：

- `+ - * / ^`
- `exp`、`ln`、`sqrt`
- `sin`、`cos`、`tan` 及反函数
- `sinh`、`cosh`、`tanh` 及反函数
- 常量：`e`、`π`、`i`、整数、有理数

所有表达式坍缩成一条文法：

```
S → 1 | eml(S, S)
```

没了。没有 `sin` 节点，没有 `mul` 节点，只有一模一样的 `eml` 节点堆成二叉树。就像只用 NAND 门搭出来的电路。

---

## 2. 这为什么不是显然的

人类早就知道 `exp` 和 `ln` 很猛。Euler 1748 年就搞出了 `e^(iθ) = cos θ + i sin θ`。Liouville 和 Ritt 围绕着 exp-log 类建立了一整套积分理论。

但关键是：**把集合压缩到两个原语，和压缩到一个，完全是两码事。**

我们早就知道 `+` 和 `×` 可以用 `exp/ln` 表达：

```
x × y = e^(ln x + ln y)
x + y = ln(e^x × e^y)
```

我们也知道三角函数可以通过复指数表达。但从来没有人把整个科学计算器压缩到**单一二元算子**加一个常量。

这篇论文真正的贡献不是某条恒等式，而是**压缩的终点居然存在**。`eml` 不是用笔推导出来的，是穷举搜出来的。作者从 36 个原语（Table 1）出发，一个一个删掉，每次检查剩下的还能不能重建完整集合。当集合卡在 `{1, eml}` 不再动弹时，他知道触到底了。

这不是一篇证明论文。这是一篇**计算发现 + 构造性验证**的论文。

## 3. 搜索到底是怎么做的

作者没有拿笔坐着推导出 `eml`。他写了一个搜索引擎。

### 消融游戏

从 36 个按钮开始。删掉一个。剩下的 35 个还能表达一切吗？能，说明被删的那个是冗余的。重复。

这产生了一个倒计时（Table 2）：

| 系统    | 原语                       | 数量  |
| ------- | -------------------------- | ----- |
| Base-36 | 完整计算器                 | 36    |
| Wolfram | `π,e,i`, `ln`, `+,×,^`     | 7     |
| Calc 3  | `exp,ln,-x,1/x,+`          | 6     |
| Calc 2  | `exp,ln,-`                 | 4     |
| Calc 1  | `e` 或 `π`, `x^y, log_x y` | 4     |
| Calc 0  | `exp`, `log_x y`           | 3     |
| **EML** | **`1`, `eml(x,y)`**        | **3** |

关键模式：每一个极小系统都包含**一对互逆函数**和**一个非交换运算**。

- `exp`/`ln` 互逆。减法不交换。
- `eml` 把两种性质熔进了一个节点。

### 数值启动技巧

直接符号验证算不动（典型 Kolmogorov 复杂度 ~7-9）。所以作者耍了个漂亮的赖：

1. 挑代数独立的超越常数，比如 `γ`（Euler-Mascheroni）和 `A`（Glaisher-Kinkelin）。
2. 在这些点上数值计算目标函数。
3. 枚举某个深度以内的所有候选 EML 表达式。
4. 如果某个候选数值匹配，对它跑反符号计算器。
5. 之后独立验证幸存者（符号化简 + 跨平台数值交叉检查）。

在 Schanuel 猜想下，这类表达式之间的巧合相等概率趋近于零。所以数值筛子是可靠的。

**这就是工程思维：** 别等闭式证明。先搭个快筛，再验证幸存者。

## 4. 工程现实

读论文是一回事。把它做成库是另一回事。以下是 `packages/emlib` 里我们怎么干的。

### 4.1 两套 AST，不是一套

我们维护两套表达式语言：

- **标准 AST**：`num`、`var`、`const`、`add`、`sub`、`mul`、`div`、`pow`、`neg`、`exp`、`ln`、`sqrt`、`sin`、`cos`……（完整的 23+ 一元 + 6 二元家族）
- **纯 EML AST**：`num`、`var`、`const`、`eml`

每个标准表达式都能 lower 成纯 EML。每棵纯 EML 树都能（有时候）lift 回可读的标准表达式。

这不是为了抽象而抽象。两种表示服务完全不同的目的：

- 标准 AST 给**人用**（解析、打印、化简）。
- 纯 EML 给**搜索用**（统一树形意味着统一搜索空间）。

### 4.2 Lowering：标准 → 纯 EML

`lower.ts` 模块实现了编译管道。本质上是带记忆化的项重写系统。

关键洞察：`reduceTypes(expr)` 不只是逐字翻译，它主动**在多个 EML 编码中选择最短的**。

比如，取负有两个 witness：

```
// 论文里的短 witness（15 个 token）
-x = E(E(1,E(1,E(1,E(E(1,1),1)))), E(x,1))

// 通用 witness（更多 token）
-x = E(E(1,E(E(E(1,E(E(1,E(1,x)),1)),E(1,E(E(1,E(E(...
```

库存了两者，通过 `chooseShortest` 选短的。乘法、除法同理。

这里工程和纯数学分道扬镳。论文证明存在性。库必须关心** token 数**，因为它决定搜索空间是否可处理。

### 4.3 Lifting：纯 EML → 标准

这是难的方向。Lowering 是确定性的。Lifting 是搜索问题。

`rewrite.ts` 模块实现了一个**成本驱动的重写引擎**：

1. **模式匹配**：模板规则如 `exp(ln(?x)+ln(?y)) → ?x*?y`
2. **贪婪归一化**：反复应用局部重写直到不动点
3. **束搜索**：贪婪卡住时探索邻居，保留 `tokenScore` 最好的
4. **精确折叠**：对常量子表达式做无损求值（有理数 + 复数运算）

评分函数刻意保持简单：

```typescript
tokenScore(expr) = tokenCount(expr) + 0.05 * typeCount(expr);
```

Token 越少越好。不同算子种类越少越好。这会把搜索偏置到人类可读的形态。

### 4.4 模式匹配：我们保留的那个 hack

重写引擎的模板写成字符串，比如 `"?x+?y"`。我们的 parser 不原生理解 `?x`，所以 `compilePattern` 做了字符串替换：

```typescript
"?x+?y" → "__h0__+__h0__" → parse() → 遍历 AST → 把对应 var 转回 hole
```

优雅吗？不。正确、可维护、而且保留了模板定义的可读性吗？是。

"纯血派"的替代方案是单独写一个 pattern parser。我们考虑过。代价要么是：

- 为 pattern 复刻整套表达式文法，或者
- 让主 parser 感知 hole，然后漏进 `lower.ts`（`reduceTypes` 不知道怎么处理 hole）

在一个模板由人写、人读、且编译频率远低于阅读频率的系统里，**字符串模板赢**。`_hN_` 前缀是藏在 `compilePattern` 内部的实现细节，从不逃逸到 pattern 模块之外。

### 4.5 合成：搜索 EML 空间

`synth.ts` 模块实现了论文的核心思想：**既然每个公式都是相同节点的二叉树，你就可以在结构上搜索这个空间。**

`synthesizePureEml(target, options)`：

1. 在采样点上计算目标表达式的值。
2. 自底向上构建 EML 树（叶子 = `1` 和变量）。
3. 用**语义指纹**（`fingerprint(values)`）去重等价树，不需要 parse。
4. 每棵树大小维护一个**束前沿**，按 MSE 和 token 数剪枝。
5. 匹配到容差以内时停止。

这是论文描述的数值启动的直接实现。区别是我们的库用 TypeScript 做复数求值，而不是 Mathematica。

## 5. 那些没人告诉你的事

### 5.1 EML 并不短

别因为 EML 简洁就用它。它并不简洁。

| 函数     | 标准形式 | EML Token 数 |
| -------- | -------- | ------------ |
| `exp(x)` | 3        | 3            |
| `ln(x)`  | 3        | 7            |
| `-x`     | 2        | 15           |
| `x*y`    | 3        | 17-25        |
| `1/2`    | 3        | 29-39        |
| `π`      | 1        | >53          |

EML 的价值是**结构统一**，不是长度。异质文法（`sin`、`+`、`^`、`log`）有几十种节点类型。EML 只有一种。这种统一性让搜索、电路映射、硬件编译变得可处理——"就用 exp 和 ln" 永远做不到这一点。

### 5.2 复数不是可选项

要得到 `i`，你需要 `ln(-1)`。要得到 `sin(x)`，你需要 Euler 公式。EML 内部在 `ℂ` 上运算，即使最终答案是实数。

这有实际后果：

- **分支切割**：`ln(z)` 在负实轴上跳 `2πi`。论文花了很多篇幅讨论这个，因为它会搞崩 naive 实现。
- **`ln(0) = -∞`**：某些 witness 依赖这个。Lean 4 把 `Complex.log 0` 设成 `0`（junk value），整个 EML 链直接断裂。
- **IEEE 754**：没问题，因为 `inf` 和带符号零是一等公民。纯 Python 会崩，因为它在 `log(0)` 上抛异常。

我们的求值器原生处理复数。如果你求值一棵会经过 `ln(-1)` 的纯 EML 树，它能跑——前提是你接受主分支。

### 5.3 "无常量 Sheffer" 仍是开放问题

NAND 能从任意输入生成 `0` 和 `1`：`NAND(x, NAND(x, x)) = 1`。EML 做不到。你**必须**在终端集合里放 distinguished constant `1`（或 `e`、或 `-∞`）。

论文明确把它列为开放问题。一个三元算子已被找到且不需要常量，但二元情况仍未解决。

## 6. 这能用来干什么

### 固定架构的符号回归

现代符号回归被文法选择折磨：

- 要不要放 `sin`？如果真实规律是 `sinh` 呢？
- 要不要放 `exp`？如果其实是 `x^2.3` 呢？
- 每缺一个算子就是一个盲区。

EML 彻底消除了这个问题。搜索空间**按构造就是完备的**：任何初等函数都在里面某处。主公式只是一棵参数化的二叉树：

```
F(x) = eml(α1 + β1*x + γ1*eml(...), α2 + β2*x + γ2*eml(...))
```

用 Adam 训练 `α, β, γ` 权重。如果它们 snap 到 `{0,1}`，你就恢复了一个精确闭式。论文报告深度 2 成功率 100%，深度 3-4 约 25%，深度 5 <1%。

优化景观很难。但它存在。

### 单指令机器

EML 表达式只是栈机上的一段 RPN 指令序列。每条指令都是同一个操作码。这意味着：

- 硬件实现极其简单（FPGA、模拟电路）
- JIT 编译 trivial
- 序列化 trivial

论文的 Table 3 明确画了这个类比：NAND 门 → 运放 → 晶体管 → EML Sheffer。

### 可解释性

神经网络找到模式时给你一堆权重。EML 树收敛时给你一个能读、能求导、能验证的公式。作者称之为"初等函数表达式的可读性"。

## 8. 代码库

这个仓库（`packages/emlib`）是用 TypeScript 从零实现论文思想的干净内核。它不是作者 Mathematica/Rust 工具包的移植。设计目标是：

- **可嵌入**：小到可以打包进 web app。
- **可 hack**：没有构建系统黑魔法。`bun test` 直接跑。
- **可扩展**：模式匹配和搜索基础设施是通用的。写字符串就能加新规则。

当前状态：

- ✅ 标准表达式 parser / printer
- ✅ 复数求值器（类 IEEE 754，支持 `inf`/`nan`）
- ✅ 精确有理数运算（`1/3 + 1/6 = 1/2`，不是 `0.5000001`）
- ✅ 23 个一元 + 6 个二元算子完整 lower 到纯 EML
- ✅ 通过模式匹配 + 束搜索的 lift / 化简
- ✅ 从数值样本合成纯 EML witness
- ✅ 可选压缩（搜索更短的等价 EML 树）
- ✅ 基于 Adam 的梯度主公式训练（`master.ts` + `train.ts`）

尚未实现：

- 分支切割感知的重写
- 完整 SI witness 数据库导入

## 9. 结论

数学通常不这么玩。我们不靠穷举搜索发现新原语。我们证明定理。

但有时候定理**就是**搜索。EML 不是推导出来的。它是先数值筛、后验证的。它居然能工作——单一算子 `exp(x) - ln(y)` 就够了——这是一个关于初等函数结构的陈述，而之前没人预料到。

如果你在造任何搜索公式的东西，这事重要。不是因为 EML 更短。而是因为 EML 是**一**。而一比三十六好得多。
